## Context

한국인공지능검증원은 Client 접수 이후 전화 상담, 견적 발송, 시험원 배정, 시험 진행, 성적서 작성, 외부 검증 사이트 업로드까지 여러 단계를 거친다. 현재 저장소는 [index.html](/Users/mjkang/Develop/ai-certi/index.html)만 존재하는 초기 상태이며, 일반적인 CRM 스타일의 대시보드 목업만 포함하고 있다.

이 스펙의 목적은 위 목업을 기반으로 실제 운영에 필요한 내부 CMS의 제품 경계, 데이터 모델, 상태 흐름, 화면 구조를 고정하는 것이다. 이 시스템은 내부 관리자와 직원이 접수 건의 현재 위치와 다음 액션을 빠르게 파악할 수 있어야 하며, 개별 기업 및 프로젝트 단위의 상세 추적과 월별 집계 모니터링을 동시에 지원해야 한다.

제품 범위는 두 영역으로 나뉜다. 첫째는 Client가 접근하는 공개 의뢰서 영역이다. 둘째는 내부 관리자와 직원이 사용하는 CMS 영역이다. 공개 의뢰서와 전화 접수는 서로 다른 유입 경로이지만 동일한 `Client` 및 `Project` 데이터 구조로 귀결되어야 한다.

## Assumptions

- [resolved][user] 내부 시스템 사용자 역할은 관리자와 직원 두 종류만 필요하다.
- [resolved][workflow] 시험원은 별도 역할이 아니라 staff 중에서 배정된 담당자로 처리한다.
- [resolved][scope] v1은 Client 포털을 제공하지 않고 공개 의뢰서 제출까지만 외부에 노출한다.
- [resolved][constraint] 기술 스택은 React, Vite, TypeScript, Convex를 기본으로 사용한다.
- [resolved][constraint] UI 원칙은 shadcn/ui를 따르되 현재 index.html의 정보 구조와 톤을 재해석한다.
- [resolved][workflow] 메일은 v1에서 초안, 발송 기록, 리마인더 중심으로 관리하고 실제 자동 발송은 필수 범위에 넣지 않는다.
- [resolved][constraint] 자료 파일은 시스템 직접 저장보다 외부 저장소 링크와 메타데이터 중심으로 관리한다.
- [resolved][workflow] 견적 금액은 모델 수와 요청사항을 참고하되 최종 금액은 사람이 수기 입력한다.
- [resolved][feedback] 직원은 전체 건을 조회할 수 있지만 수정은 담당 범위 중심으로 제한한다.
- [resolved][context] 현재 저장소는 제품 구현이 시작되기 전 단계이며 문서가 구현 기준을 제공해야 한다.

## Decisions

### 1. 제품 구조

- 하나의 제품 안에 공개 의뢰서 라우트와 내부 CMS 라우트를 함께 둔다.
- 공개 의뢰서는 신규 `Client` 및 `Project` 생성의 외부 입력 채널이다.
- 전화 문의는 내부 CMS에서 수기 등록하는 별도 입력 채널이다.

### 2. 핵심 엔티티

- `User`
  - 역할: `admin | staff`
  - 내부 로그인 사용자만 포함한다.
- `Client`
  - 기업명
  - 담당자 성명
  - 이메일
  - 연락처
  - 사업자등록번호(optional)
- `Project`
  - 회사명
  - 시험대상 설명
  - 시험결과서 발급 요청일
  - 기타 요청사항
  - 유입경로: `web | phone`
  - 담당자
  - 시험원
  - 현재 상태
- `ConsultationNote`
  - 전화상담 메모
  - 작성 시각
  - 작성자
- `Quote`
  - 모델 수
  - 요청사항 요약
  - 수기 견적 금액
  - 발송일
  - 회신기한
  - 팔로업 상태
- `ArtifactLink`
  - 자료 유형
  - 외부 링크
  - 설명
  - 업로더
  - 업로드일
- `ReportWorkflow`
  - 초안
  - 내부검토
  - 고객검토
  - 최종확정
- `G4vTracking`
  - 업로드 여부
  - 업로드일
  - 검증 상태
  - 이슈 메모
- `TaskReminder`
  - 팔로업
  - 검토 요청
  - 미완료 체크

### 3. 관계 및 타입 기준

- `Role = 'admin' | 'staff'`
- `ProjectStatus = '접수' | '상담중' | '견적/신청서 발송' | '회신대기' | '시험원배정' | '시험협의중' | '시험진행중' | '성적서초안' | '내부검토' | '고객검토' | '최종확정' | 'g4v업로드' | '완료'`
- `IntakeSource = 'web' | 'phone'`
- `QuoteStatus`는 초안, 발송완료, 회신대기, 팔로업필요로 정의한다.
- `ReportStatus`는 초안, 내부검토, 고객검토, 최종확정으로 정의한다.
- `G4vStatus`는 미업로드, 업로드완료, 검증중, 이슈발생, 완료로 정의한다.
- `Client 1:N Project`
- `Project 1:N ConsultationNote`
- `Project 1:N ArtifactLink`
- `Project 1:N TaskReminder`

### 4. 표준 상태머신

모든 Project는 아래 상태 흐름을 기본으로 사용한다.

1. `접수`
2. `상담중`
3. `견적/신청서 발송`
4. `회신대기`
5. `시험원배정`
6. `시험협의중`
7. `시험진행중`
8. `성적서초안`
9. `내부검토`
10. `고객검토`
11. `최종확정`
12. `g4v업로드`
13. `완료`

상태는 단순 표시값이 아니라 운영자가 현재 단계와 다음 액션을 해석할 수 있게 하는 기준이다. 세부 진행 항목인 견적 발송, 자료 적재, 검토 요청, g4v 이슈는 보조 엔티티와 리마인더로 관리하되, 메인 진행상태는 위 상태집합으로 통일한다.

  - 담당자 필터

### 6. 알림과 후속 조치

- 견적 및 신청서 발송 기록이 저장되면 기준 시점을 남긴다.
- 회신이 없는 상태로 3일이 지나면 `TaskReminder`를 생성한다.
- v1의 기본 알림 표면은 앱 내 대시보드, 리스트 배지, 상세 화면 태스크 영역이다.
- 내부 사용자 이메일 알림과 실제 메일 자동 발송은 후속 단계로 미룬다.

### 7. 디자인 및 정보 구조

- 현재 [index.html](/Users/mjkang/Develop/ai-certi/index.html)의 요약 메트릭, 테이블, 액션 카드 구조를 제품 대시보드의 출발점으로 사용한다.
- 시각 스타일은 shadcn/ui의 컴포넌트 원칙과 일관된 spacing, 상태 표현, 카드 및 테이블 패턴으로 재구성한다.
- 목표는 단순한 CRM 복제가 아니라, 시험 진행상황 추적에 맞는 운영형 대시보드로 재매핑하는 것이다.

## Risks

- 실제 운영 과정에서 상태 정의가 더 세분화되어야 할 가능성이 있다.
- 견적 금액이 항상 수기 입력이면 향후 반복 업무가 늘어날 수 있다.
- 파일을 링크 중심으로 관리할 경우 외부 저장소 접근 정책과 링크 만료 정책이 별도로 필요할 수 있다.
- g4v 외부 사이트의 실제 처리 흐름이 현재 서술보다 복잡할 경우 추가 상태나 이슈 모델이 필요할 수 있다.
- staff의 전체 조회와 담당 중심 수정 권한을 어느 수준까지 UI에서 제한할지는 구현 단계에서 더 구체화해야 한다.

## Notes

- 이 문서는 제품 구현 전 기준 문서이며, 개발자가 바로 착수할 수 있는 수준의 도메인 모델과 화면 경계를 제공하는 것을 목표로 한다.
- 저장소가 초기 상태이므로 `index.html`은 최종 산출물이 아니라 정보 구조 참고 자산으로 본다.
- 추천 초기 라우트 구조는 공개 의뢰서 영역과 내부 CMS 영역을 분리한 형태다.
- 추천 초기 구현 우선순위는 공개 의뢰서, 내부 등록, 목록/상세, 상태 전이, 대시보드 집계 순서다.
