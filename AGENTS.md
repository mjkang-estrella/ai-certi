# AGENTS.md

이 저장소의 기본 작업 언어는 한국어다. 문서, 이슈 정리, UI 문구, 도메인 용어, 테스트 시나리오는 특별한 이유가 없으면 한국어로 작성한다.

## 제품 맥락

- 이 제품은 KAIC CMS다.
- 목적은 접수부터 시험 완료까지의 진행상황을 내부에서 추적하고 관리하는 것이다.
- 주요 사용자는 내부 관리자와 직원이다.
- Client는 공개 의뢰서 페이지를 통해 유입되며, 내부 CMS에는 로그인하지 않는다.

## 기술 기준

- 프론트엔드 기본 스택은 `React + Vite + TypeScript`를 사용한다.
- 백엔드는 `Convex`를 단일 백엔드로 사용한다.
- UI 기본 원리는 `shadcn/ui`를 따른다.
- 디자인 레이어는 현재 저장소의 `index.html`이 가진 정보 구조와 분위기를 재해석해 반영한다.
- Tailwind 기반 구성을 기본으로 하되, 작은 범위의 커스텀 스타일은 허용한다.

## 구현 원칙

- 단순한 해결책을 우선한다.
- TypeScript 타입을 먼저 설계하고, 데이터 구조와 상태를 명시적으로 유지한다.
- 작은 API와 읽기 쉬운 이름을 선호한다.
- 불필요한 추상화, 과한 레이어 분리, 조기 최적화는 피한다.
- 작은 기능에 무거운 라이브러리를 추가하지 않는다.
- 다른 강한 엔지니어가 빠르게 읽고 수정할 수 있는 코드를 작성한다.
- 암묵적 동작보다 명시적 동작을 우선한다.

## 도메인 모델 원칙

- `Client`와 `Project`는 분리된 엔티티로 다룬다.
- 하나의 `Client`는 여러 `Project`를 가질 수 있다.
- 진행상황은 단순 완료 여부가 아니라 상세 파이프라인 상태로 관리한다.
- 파일은 시스템 내 직접 저장보다 `링크 + 메타데이터` 중심으로 관리한다.
- 메일 기능은 v1에서 `초안 작성`, `발송 기록`, `리마인더` 중심으로 다룬다.
- 견적 금액은 v1에서 규칙 계산보다 수기 입력을 우선한다.

## 기본 권한 원칙

- 내부 사용자 역할은 `admin`, `staff` 두 종류만 사용한다.
- `admin`은 전체 현황과 직원별 진행상황을 모두 확인하고 관리할 수 있다.
- `staff`는 전체 건을 조회할 수 있다.
- `staff`의 수정 권한은 본인 담당 건 중심으로 설계한다.
- 시험원은 별도 역할이 아니라 `staff` 중 배정된 담당자로 처리한다.

## 핵심 워크플로우 원칙

- 공개 의뢰서 제출 또는 전화 접수로 `Client`와 `Project`가 생성된다.
- 전화 상담 내용은 메모와 이력으로 남긴다.
- 견적서와 정식 실험 신청 의뢰서 발송 여부를 기록한다.
- 발송 후 3일 동안 답변이 없으면 앱 내 리마인더를 생성한다.
- 시험원 배정 후 시험 협의, 시험 진행, 성적서 작성, `g4v` 업로드, 완료까지 상태를 추적한다.

13. `완료`

## UX 원칙

- 한국어 UI를 기본으로 한다.
- 관리자 대시보드와 담당자 업무 화면을 모두 고려한다.
- 월별 현황, 상태별 카운트, 직원별 진행상황이 한눈에 보여야 한다.
- `index.html`의 대시보드 구성을 버리지 말고, shadcn/ui 컴포넌트 구조로 옮긴다.
- 보기 좋은 것보다 운영자가 빠르게 파악하고 갱신할 수 있는 UX를 우선한다.

## 작업 가이드

- 새로운 기능을 만들 때는 먼저 이 제품의 상태머신과 엔티티 관계를 해치지 않는지 확인한다.
- 공개 의뢰서와 내부 CMS는 같은 제품 안의 다른 라우트로 설계한다.
- 파일 업로드가 꼭 필요하지 않다면 링크 저장으로 끝내는 방식을 먼저 검토한다.
- 자동화가 필요해 보여도 v1에서는 사람이 최종 판단하는 흐름을 유지한다.
- 화면을 추가할 때는 관리자용 집계 화면과 개별 프로젝트 상세 화면 중 어디에 속하는지 먼저 분명히 한다.
